# M1 Frozen Contract — Task Spine + Kanban Board

**Status:** FROZEN. Two parallel agents implement against this; neither edits the
other's directory. The orchestrator does the integration pass.

M1 goal: a persistent task registry with a Kanban board. Create tasks, run one
through the sidecar, watch it stream, see its status transition and land in Done.
Serial execution (one run at a time) — the auto-loop + concurrency are M2.

## Ownership (disjoint)

- **Agent A — Rust core:** `apps/desktop/src-tauri/**` ONLY.
- **Agent B — Web board:** `apps/web/**` ONLY.
- Neither touches `apps/sidecar`, `packages/*`, root config, or the other's dir.
- The sidecar needs NO changes — its `SessionManager` already multiplexes
  sessions; M1 correlation is trivial because execution is serial.

## Task model (the shared shape)

Rust serde struct with `#[serde(rename_all = "camelCase")]`; TS mirror is
identical field names. Persisted as JSON.

```
Task {
  id:           string        // uuid v4, generated by the core on create
  title:        string
  description:  string        // may be empty
  status:       TaskStatus
  dependencies: string[]      // task ids; stored in M1, enforced in M2
  model:        string | null // null = core/config default
  createdAt:    number        // epoch ms
  updatedAt:    number        // epoch ms
  sessionId:    number | null // sidecar session id of the last/current run
  summary:      string | null // result text on success
  error:        string | null // message on failure
  costUsd:      number | null // cost of the last run
}

TaskStatus = "backlog" | "ready" | "in_progress" | "waiting_approval" | "done" | "failed"
```

For M1, `ready` and `waiting_approval` are reserved (defined, not yet produced).
A new task is `backlog`. `run_task` → `in_progress` → `done` | `failed`.

## Tauri commands (Rust `#[tauri::command]`, snake_case)

All return `Result<T, String>`. Args arrive camelCase from the TS bridge.

```
list_tasks()                                   -> Vec<Task>
create_task(title: String, description: String) -> Task        // status=backlog
update_task(id: String, patch: TaskPatch)      -> Task         // partial update
delete_task(id: String)                        -> ()
run_task(id: String)                           -> ()           // see below
cancel_task(id: String)                        -> ()           // interrupt current run (best-effort)

TaskPatch { title?, description?, status?, dependencies?, model? }  // all optional
```

`run_task` behavior:
1. Set the task `in_progress` (persist + emit `nc:task`).
2. Ensure the persistent sidecar is running (spawn lazily if needed).
3. Build the prompt: `title` + (description non-empty ? "\n\n" + description : "").
4. Send `{"type":"start-session","prompt":<built>,"model":<task.model ?? null>}`.
5. Mark this the active task; relay every sidecar event as `nc:session`.
6. On `session-completed`: status=`done`, set `summary`=result, `costUsd`, `sessionId`; persist + emit `nc:task`.
   On `session-failed`: status=`failed`, set `error`; persist + emit `nc:task`.
7. M1 serial guard: if a task is already `in_progress`, `run_task` returns an
   `Err("a task is already running")` (no queue yet — that's M2).

Permission requests during a run are auto-denied (M1, same as M0).

## Events (Rust → webview, via `app.emit`)

```
"nc:task"     payload = Task            // emitted on every create/update/status change; UI upserts by id
"nc:session"  payload = { taskId: string, event: NightcoreEvent }
                                        // streamed session events for the running task
```

`NightcoreEvent` is forwarded verbatim from the sidecar (the existing engine
event union: session-started/ready, assistant-delta {text,partial},
tool-use-requested {toolName,input}, tool-result, session-completed
{costUsd,numTurns,durationMs,usage}, session-failed {reason,message},
session-status). The web side only needs to render: assistant-delta (with the
partial-dedup rule from M0's App), tool-use-requested, session-completed,
session-failed.

## Persistence (Rust core owns it)

- Dir: `<project>/.nightcore/tasks/` where `<project>` = the workspace root
  (same cwd resolution M0 uses: `CARGO_MANIFEST_DIR/../../..`). Create if missing.
- One file per task: `<project>/.nightcore/tasks/<id>.json` (pretty JSON).
- Load all on startup into in-memory state (managed Tauri state, `Mutex`/`RwLock`).
- Write the file on every create/update; remove it on delete. `.nightcore/` is
  already gitignored.

## Sidecar lifecycle (Rust core)

- ONE persistent sidecar (`bun run apps/sidecar/src/index.ts`, cwd = workspace
  root), spawned lazily on first `run_task`, kept alive in managed state.
- Keep its `stdin` writer (send commands) and spawn a reader task over `stdout`
  that parses each NDJSON line into a `serde_json::Value` and emits `nc:session`
  with the current active `taskId`, applying the status transition on terminal
  events.
- M0's `start_prompt` command + `nc:event` channel are REMOVED (superseded).

## Web board (Agent B)

Replace M0's single-prompt `App` with a Kanban board:

- Columns grouped by status: **Backlog** (backlog, ready) · **In Progress**
  (in_progress, waiting_approval) · **Done** (done) · **Failed** (failed).
- Cards show: title, 1–2 line description, status dot, cost when present.
- "New task" affordance (title + description) → `create_task`.
- Click a card → detail panel/drawer: description, a **Run** button (`run_task`,
  disabled while any task is in_progress), live streamed output assembled from
  `nc:session` for that taskId (reuse M0's partial-dedup), and status/cost.
- Subscribe once to `nc:task` (upsert board state by id) and `nc:session`
  (append to the matching task's live stream).
- Keep the dark aesthetic, Tailwind v4, and the `bridge.ts` invoke/listen +
  `isTauri()` patterns already established. Extend `bridge.ts` with the new
  commands + a `Task`/`TaskStatus` TS type mirroring this contract exactly, and
  validate event payloads defensively.
- Must build green: `bun run --filter @nightcore/web build`.

## Definition of done (integration)

- `cargo check` green; `bun run --filter @nightcore/web build` green.
- Create a task → it appears in Backlog → Run → streams in the detail panel →
  lands in Done with a cost; a deliberately failing run lands in Failed.
- All UI mutations go through Tauri commands; all board updates arrive via
  `nc:task`/`nc:session` (no direct state coupling) — the boundary holds.
